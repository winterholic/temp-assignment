# 이미지 처리 비동기 작업 서버

---

## 실행 방법

### Docker로 실행
로컬 컨테이너 환경에서 실행 가능하도록 docker-compose를 사용하여 실행할 수 있도록 인프라를 구성하였습니다.<br>
docker를 사용해서 바로 실행 가능합니다.

```bash
docker-compose up --build
```

서버가 뜨면 `http://localhost:8080` 에서 사용할 수 있습니다.<br>
스웨거는 `http://localhost:8080/swagger-ui/index.html#/` 에서 확인할 수 있습니다.

---

## 엔드포인트

| 경로 | 설명 |
|------|------|
| `POST /jobs` | 이미지 처리 요청 |
| `GET /jobs/{jobId}` | 작업 상태 조회 |
| `GET /jobs` | 전체 작업 목록 |

---

## 상태 모델

```
PENDING → PROCESSING → COMPLETED
                     ↘ FAILED
```

- PENDING: 초기 상태입니다. Mock Worker에 전달하기 이전 상태로 설정하였습니다.
- PROCESSING: Mock Worker에 전달하여 처리중인 상태입니다.
- COMPLETED: 처리가 완료되어 result에 결과값이 나온 상태입니다.
- FAILED: 처리 실패를 응답받은 상태입니다.

기존 Mock Worker가 가지는 상태에 PENDING을 추가하였습니다.

---

## 설계 설명



### 중복 요청 처리

만약 동일한 imageUrl로 요청이 들어온 경우에 PENDING또는 PROCESSING상태의 작업이 있다면 해당 작업의 id를 반환하고 COMPLETED나 FAILED상태인 경우는 작업을 다시 생성합니다.

### 실패 처리 전략

- 429: Mock Worker가 너무 많은 작업을 시도중이므로 처리를 거부한 상태이기 때문에, 일정 간격으로 재시도
- 5xx: 최대 3회 재시도
- 폴링 타임아웃: 10분 초과 시 FAILED 처리

### 처리 보장 모델

최소한번 처리 보장 모델을 지향합니다. MockWorker는 과부하가 많은 작업을 처리중이기 때문에, 정확히 한번 처리로는 문제의 요구사항을 충족 시킬 수 없으며, 중복을 처리하는 시스템을 따로 두었기 때문에 굳이 최대 한번 처리를 할 이유가 없다고 판단하였습니다.

### 동시 요청

동시에 같은 imageUrl이 요청으로 들어온다면 DB 트렌젝션과 중복 요청 처리 로직에 따라 문제가 일어나지 않을 것으로 판단했습니다.

### 트래픽 증가 시 병목 지점

- 스레드 풀: 각 job이 스레드를 점유하며 최대 10분간 폴링하므로 동시 처리 수에 한계가 있습니다.
- H2 파일 DB: 단일 JVM 전용이라 수평 확장이 불가합니다. 실제 운영이라면 다른 DB엔진을 사용해야 합니다.

### 외부 시스템 연동 방식 및 선택 이유
Mock Worker가 무거운 작업을 실행하여 비동기로 처리하는 구조이므로, 그에 맞춰서 위 서버도 그 상태를 지속적으로 확인해주는 비동기로 폴링하는 방식을 선택했습니다.
